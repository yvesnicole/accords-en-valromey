## Build & Dev

```bash
npm run dev      # tinacms dev -c "astro dev"  → localhost:4321
npm run build    # tinacms build --content=local --skip-cloud-checks && astro build
npm run preview  # astro preview (serve dist/)
npx astro check  # type checking
npm run reindex  # force Tina Cloud content reindex (pushes to main)
```

**Build gotchas**:
- `--skip-cloud-checks` is REQUIRED for local/CI builds without Tina Cloud
- `tina/__generated__/` MUST be committed to git (required for Tina Cloud schema detection)
- `tina/tina-lock.json` must be committed (generated by `tinacms dev`, not `tinacms build`)

## Environment Variables

```
NEXT_PUBLIC_TINA_CLIENT_ID=   # Tina Cloud client ID (public, embedded in build)
TINA_TOKEN=              # Tina Cloud token (secret, build-time only)
```

Both are set as Netlify environment variables (Site settings → Environment variables). `NEXT_PUBLIC_TINA_CLIENT_ID` is public (embedded in build), `TINA_TOKEN` is build-time only. Both are also hardcoded in `tina/__generated__/` by `tinacms dev` — this is expected and not a security risk since they are non-secret values required for the admin UI.

## Content Architecture

**TinaCMS owns all content.** Astro's content layer is a stub (`src/content.config.ts` loads a non-existent `.json` pattern just to suppress warnings about orphaned directories).

### Content Collections (defined in `tina/config.ts`)

| Collection | Path | Fields |
|---|---|---|
| `concert` | `src/content/concerts/{fr,en}/*.mdx` | title, date, locale, translationKey, program (rich-text, isBody), description, image, helloAssoLink, mapsLink, mapsLabel, ticketLabel |
| `musician` | `src/content/musicians/{fr,en}/*.mdx` | name, locale, translationKey, instrument, photo, photoCredit, bio (rich-text, isBody) |
| `festival` | `src/content/festival/{fr,en}/*.mdx` | title, locale, translationKey, subtitle, image, imageCaption, body (rich-text, isBody) |
| `homepage` | `src/content/homepage/{fr,en}/*.mdx` | title, locale, translationKey, heroTitle, heroSubtitle, introTitle, introDescription, nextConcertCtaLabel |
| `ticketInfo` | `src/content/ticket-info/{fr,en}/*.mdx` | title, locale, translationKey, sectionTitle, sectionSubtitle, discountText, email, cards (repeater: title + text) |
| `contactInfo` | `src/content/contact-info/{fr,en}/*.mdx` | title, locale, translationKey, email, address, phone, mapEmbedUrl, facebookUrl, instagramUrl |
| `supportInfo` | `src/content/support-info/{fr,en}/*.mdx` | title, locale, translationKey, subtitle, membershipUrl, donationUrl, membershipTitle, membershipText, donationTitle, donationText |

### Bilingual Content Pattern

Every logical document has **two MDX files** (one per locale) linked by a shared `translationKey`:

```
src/content/concerts/
  fr/opening-2026.mdx    → locale: "fr", translationKey: "opening-2026"
  en/opening-2026.mdx    → locale: "en", translationKey: "opening-2026"
```

**Data fetching** uses `src/lib/data.ts` which reads local MDX files via `gray-matter`:
```typescript
import { getConcerts, getConcert, getMusicians, getFestival, getHomepage, getTicketInfo, getContactInfo, getSupportInfo } from '../lib/data';
const concerts = await getConcerts('fr'); // auto-sorted by date
const festival = await getFestival('fr');
const homepage = await getHomepage('fr');
```

## i18n System

### How It Works

- **FR is default** (root path `/`), **EN is under `/en/`**
- **Parallel page files**: `src/pages/billetterie.astro` (FR) and `src/pages/en/tickets.astro` (EN)
- Route slugs are mapped in `src/i18n/utils.ts` → `translateSlug()`
- UI strings in `src/i18n/ui.ts` → `ui.fr` / `ui.en` dictionaries (36 keys)
- Language switcher in `Header.astro` + `LanguageSwitcher.astro`

### Adding a New i18n Key

1. Add to `src/i18n/ui.ts` under both `fr` and `en` objects
2. TypeScript auto-infers the key via `type UiKey = keyof typeof ui.en`
3. Use in pages: `const t = useTranslations(locale); t('my.new.key')`

### Current i18n Issues

## Deployment

- **Platform**: Netlify (auto-deploy on push to `main`)
- **Config**: `netlify.toml` — Node 22, build command `npm run build`, publish `dist/`
- **Caching**: `/_astro/` assets cached forever (immutable), static assets 30 days
- **Security**: X-Frame-Options SAMEORIGIN, X-Content-Type-Options nosniff
- **SSL**: Auto-managed by Netlify

### Environment Variables (Netlify)

Set in Netlify Dashboard → Site settings → Environment variables:

- `NEXT_PUBLIC_TINA_CLIENT_ID` — Tina Cloud client ID (public, embedded in build)
- `TINA_TOKEN` — Tina Cloud token (secret, build-time only)
- `REINDEX_GITHUB_TOKEN` — GitHub PAT with repo scope (secret, used by the auto-reindex plugin to push commits)

### Auto-Deploy Flow

```
TinaCMS Admin edit → commit to GitHub → Netlify auto-build → site updated
                                                    ↓
                                    Auto-reindex plugin pushes timestamp
                                    to tina-lock.json → Tina Cloud reindexes
```

Every push to `main` triggers an automatic build and deploy. The `tina-reindex` Netlify plugin also pushes a timestamp to `tina-lock.json` after each build, forcing Tina Cloud to reindex content and preventing ghost folders in the admin UI.

## Common Gotchas

1. **`trailingSlash: 'always'`** — All internal links MUST end with `/`. Missing trailing slash = broken link in production.
2. **`content.config.ts` is a stub** — Don't use `astro:content` to query data. Use `src/lib/data.ts` which reads local MDX files via `gray-matter`.
3. **EN pages have their own `.astro` files** — They're NOT dynamically generated. `src/pages/en/tickets.astro` is a separate file from `src/pages/billetterie.astro`.
4. **Tailwind v4 `@theme`** — Custom tokens are defined in `src/styles/global.css`, NOT in a config file.
5. **Tina `tina-lock.json`** — Commit it. If missing, run `tinacms dev` first to generate it.
6. **`@/` alias** — Maps to `src/`. Use `import X from '@/components/...'`.
7. **HelloAsso links** — External ticketing platform. Links are stored as `helloAssoLink` field on concerts. Don't build a custom checkout.
8. **Netlify auto-deploys** — Every push to `main` triggers a build. Set `NEXT_PUBLIC_TINA_CLIENT_ID` and `TINA_TOKEN` in Netlify Dashboard → Site settings → Environment variables.
9. **Ghost folders in admin UI** — The `tina-reindex` Netlify plugin automatically pushes a reindex commit after each deploy. If ghost folders persist, run `npm run reindex` manually. The plugin uses `REINDEX_GITHUB_TOKEN` (set in Netlify env vars) and skips itself if the last commit already contains `[tina-reindex]` to prevent infinite loops.

## When Adding Features

1. **New page?** → Create FR `.astro` + EN `.astro` + nav entry in Header + route mapping (if slug differs) + i18n keys + optional TinaCMS content
2. **New TinaCMS field?** → Edit `tina/config.ts`, run `tinacms dev` to regenerate types, commit `tina-lock.json`
3. **New component?** → Follow existing patterns: `interface Props`, pure Astro (no React), inline `<script>` for vanilla JS interactivity
4. **New i18n key?** → Add to both `fr` and `en` in `src/i18n/ui.ts`, use `t('key.name')` in pages
5. **Styling?** → Use Tailwind utility classes + custom tokens (`bg-sage`, `text-ivory`, `font-serif`, `rounded-card`, `py-section`)
